2. 核心概念：模块化布局与组件复用
在开始写代码之前，我们先理解一个重要的设计理念：模块化布局。

2.1 为什么需要统一界面？
如果每个算法都单独做一个界面，不仅开发量巨大，而且用户体验割裂。想象一下，每次使用不同算法都要重新学习操作方式，多累呀！

统一界面的好处：

一致性：所有算法共用相同的操作流程（选择→输入→运行→查看结果）。
可维护性：修改一次界面布局，所有算法都受益。
扩展性：未来添加新算法时，只需关注算法逻辑，界面框架无需大改。
2.2 界面布局拆解
我们的主界面将分为四个功能区，就像一张“操作地图”：

算法选择区：一个下拉菜单（Combobox），列出所有支持的算法，用户点击后切换。
数据输入区：一个文本输入框，允许用户手动输入数据（如排序数组、图节点等），旁边有“随机生成”按钮。
过程展示区：一个大的画布（Canvas），用于绘制排序动画、图路径等。
结果信息面板：显示算法执行过程中的步骤、最终结果、时间复杂度等信息。

def create_control_frame(self):
    control_frame = ttk.Frame(self.root)
    control_frame.pack(side="top", fill="x", padx=10, pady=5)
    
    # 算法选择标签
    ttk.Label(control_frame, text="选择算法:").pack(side="left", padx=5)
    
    # 下拉菜单（Combobox）
    self.algo_var = tk.StringVar()
    self.algo_combobox = ttk.Combobox(
        control_frame, 
        textvariable=self.algo_var,
        values=["快速排序", "Dijkstra算法", "汉诺塔"],  # 后续可扩展
        state="readonly"  # 只读，防止用户输入
    )
    self.algo_combobox.pack(side="left", padx=5)
    self.algo_combobox.set("快速排序")  # 默认选项
    
    # 确认按钮
    ttk.Button(control_frame, text="确认选择", command=self.on_algo_select).pack(side="left", padx=5)
关键点：

state="readonly" 防止用户输入无效值。
self.algo_var 用于获取当前选中的算法名称。
self.on_algo_select 是一个回调函数，当用户点击“确认选择”时触发。
⚠️ 注意：现在 on_algo_select 方法还没有定义，我们会在后续实现，暂时可以留空或用 pass。

def create_canvas_frame(self):
    canvas_frame = ttk.Frame(self.root)
    canvas_frame.pack(expand=True, fill="both", padx=10, pady=5)
    
    self.canvas = tk.Canvas(canvas_frame, bg="white")
    self.canvas.pack(expand=True, fill="both")

def create_info_frame(self):
    info_frame = ttk.Frame(self.root)
    info_frame.pack(side="bottom", fill="x", padx=10, pady=5)
    
    # 步骤信息
    ttk.Label(info_frame, text="步骤:").pack(anchor="w")
    self.steps_text = tk.Text(info_frame, height=5, wrap="word")
    self.steps_text.pack(fill="x")
    
    # 复杂度信息
    self.complexity_label = ttk.Label(info_frame, text="时间复杂度: 待计算 | 空间复杂度: 待计算")
    self.complexity_label.pack(anchor="w")
提示：

画布 Canvas 是后续绘制动画的核心，我们会在第2章、第3章中大量使用它。
steps_text 是一个 Text 组件，可以显示多行文本，用于记录算法执行过程中的每一步。
complexity_label 用于显示算法的时间/空间复杂度。

5. 扩展思考：如何做到真正的“统一”？
我们目前只实现了界面的“外壳”。真正的统一界面，还需要做到：

统一的数据接口：无论哪种算法，输入数据都通过 self.data_entry 获取，输出都通过 self.canvas 和 self.steps_text 展示。
统一的执行流程：所有算法都遵循“选择→输入→运行→展示”的流程。
动态切换：当用户切换算法时，输入框的提示信息（如“请输入数组”或“请输入图结构”）应随之改变。
💡 进阶挑战：尝试在 on_algo_select 方法中，根据选中的算法动态修改 self.data_entry 的提示文本（使用 insert 和 delete 方法）。
